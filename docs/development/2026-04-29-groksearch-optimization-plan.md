# GrokSearch 启动稳定性与安装路径优化计划

**日期**：2026年04月29日
**状态**：进行中
**适用仓库**：`AoManoh/GrokSearch`

## 背景

近期反馈集中在 `uvx --from git+https://github.com/AoManoh/GrokSearch@main grok-search` 的 `stdio` 初始化阶段，典型表现为：

- `MCP Server Error: failed to initialize server: transport error: transport closed`
- `timeout 60s`

现阶段已经确认：问题重点不在搜索运行时本身，而在首次安装、模块导入和 MCP 子进程建立握手之前的初始化路径。

同时，本仓库由当前维护团队独立维护，后续优化以当前 fork 的发布节奏、验证结论和使用场景为准，不以向原作者仓库提交 PR 作为默认目标。

## 已确认事实

### 1. 当前问题主要发生在启动阶段

- `GrokSearch` 在成功启动后，`web_search`、`web_fetch`、`web_map` 等功能可以正常工作。
- 现有验收结果表明，启动后的运行链路和上游搜索能力并不是本轮主要瓶颈。
- 因此当前优化重点应放在 `stdio` 入口、安装路径和导入成本，而不是先改动搜索结果语义。

### 2. 第一轮启动减负已经完成

当前工作区已经完成以下基础优化：

- 将 HTTP 相关应用构造从 `server.py` 的顶层导入路径中移出。
- 将 `server.app` 改为懒加载，保留 HTTP 兼容访问方式。
- 删除 `pyproject.toml` 中重复声明的直接依赖：`mcp[cli]`、`starlette`、`uvicorn`。
- 同步 `uv.lock` 与 `egg-info` 元数据。
- 增补回归测试，确认 `server.app` 与 HTTP 路由兼容性未被破坏。

### 3. 仍然存在的主要风险

- `fastmcp` 自身导入链较重，仍是当前导入期开销的大头。
- `uvx --from git+https://...@main` 依赖 Git 网络、缓存命中和分支解析，首启抖动无法只靠本地 import 优化完全消除。
- 当前 fork 还没有稳定 tag 或 release，默认文档仍容易把用户引向移动目标 `@main`。

## 优化目标

### 目标一：降低 `stdio` 初始化超时概率

尽可能减少首次安装和模块导入的仓库侧额外负担，优先让 MCP 子进程在常见 Windows 使用环境下更稳定地完成初始化握手。

### 目标二：建立稳定安装锚点

避免默认把用户引向 `@main`，逐步把安装和导出配置切换到固定 tag 或固定 commit，降低远端分支变化对初始化稳定性的影响。

### 目标三：为后续优化建立可量化基线

后续所有启动优化都应配套记录测量方法、导入时间和冷启动验收结果，避免凭主观体感判断“快了没有”。

## 非目标

- 不把“建议用户改客户端超时配置”视为本任务的完成标准。
- 不在本任务中把 `GrokSearch` 改造成与当前功能定位不同的产品。
- 不以向原作者仓库发起 PR 作为计划出口。

- 从默认依赖中移除重复声明的 HTTP/CLI 直接依赖。

3. 最后执行 Phase 4，用真实客户端复核优化效果。
